git, from first principles

How worktrees & branches actually work

Built entirely from the overnight-queue run you just watched in your own repo — every commit, branch, and cleanup step below is the real thing that happened, not a made-up example.

Three ideas you need before any of this makes sense

Everything else on this page is just combinations of these three things.

1

Commit

A permanent, numbered snapshot of your files. Once made, it never changes — e.g. 90c6c61 is the exact snapshot after HELLO.md was added.

2

Branch

Not a copy of your files. Just a movable sticky note with a name on it, pointing at one commit. Making a new commit just moves that branch's sticky note forward.

3

HEAD

A pointer to whichever branch you currently have checked out — in other words, what your working folder shows you right now.

The rule git normally enforces

One repo → one working folder → one branch checked out at a time. If you git checkout other-branch, the files on disk in your project folder change to match it. Only one branch's files can be sitting in your folder at once.

The problem: overnight-queue can't obey that rule

It needs an AI agent to make file changes and commits in the background, without ever touching the branch and files you currently have open. Checking out a different branch in your normal project folder would yank your own files out from under you mid-session. The fix is a git worktree: a second, fully independent working folder, checked out on a different branch, sharing the exact same underlying .git database.

Your normal checkout

project_template/
├── .git/            ← the actual database
├── queue/
├── docker-sandbox/
└── ...your files...
checked out on: your-branch

The worktree, alongside it

project_template/
├── .git/            ← same database, shared
└── .worktrees/
    └── overnight-queue/
        ├── .git       ← pointer file, not real folder
        └── ...a second full copy...
checked out on: overnight-queue/<timestamp>

Both folders are real, both are fully usable at the same time — but there is still only one .git database underneath them both. Commits made in the worktree are just as real and permanent as commits made in your normal folder.

What actually happened in your run

From /overnight-queue run until +5 minutes to the commit landing.

  • A new branch was created: overnight-queue/20260703-132856, pointing at whatever commit was HEAD at that moment (56b6192).
  • A worktree was created: a second checkout at .worktrees/overnight-queue/, checked out on that brand-new branch — your own project folder was never touched.
  • Claude ran hello.md's instructions inside that worktree, creating HELLO.md there — only in the worktree, nowhere else.
  • commit_task() ran git add -A && git commit inside the worktree, creating commit 90c6c61 on the overnight-queue/20260703-132856 branch.
  • hello.md moved to queue/done/, the queue was empty, so the run exited immediately instead of waiting out the rest of the 5 minutes.
  • Cleanup removed the worktree folder and deleted the branch pointer — covered in detail below.

How to inspect any of this yourself

See every branch that exists:

git branch -a
* Make_overnight-queue_agent_agnostic
  main
  remotes/origin/main

See every worktree that exists right now:

git worktree list
/Users/.../project_template   56b6192 [Make_overnight-queue_agent_agnostic]

See a branch's history — works even after its worktree is deleted, because the commit lives in .git, not in the worktree folder: 

git log --oneline overnight-queue/20260703-132856
90c6c61 hello.md: completed by overnight-queue
56b6192 Self-heal stale worktrees and skip pacing sleep...

Read a file's exact content on that branch, without checking it out:

git show overnight-queue/20260703-132856:HELLO.md
hello

The gotcha you actually hit earlier

The worktree's .git "file" isn't a real git folder — it's a one-line pointer. Inside .worktrees/overnight-queue/.git it just says: 
cat .worktrees/overnight-queue/.git
gitdir: /workspace/.git/worktrees/overnight-queue

/workspace is where the project is mounted inside the Docker container that runs the queue. From your Mac's own terminal, /workspace doesn't exist — so any git command run against that worktree directly from the host fails with fatal: not a git repository. That's exactly the error you saw. The fix wasn't to avoid worktrees — it was to make queue-runner.sh check whether an existing worktree is actually still valid before reusing it, and rebuild it if not.

Why deleting the worktree & branch was safe

git worktree remove

Deletes the second working folder and deregisters it. The .git database itself — and every commit already made — is untouched.

git branch -D

Removes the sticky note (the branch name). It does not delete the commit it was pointing to — it just stops pointing at it.

A commit stays in .git as long as something still points to it. Once the last branch pointing to a commit is deleted, it becomes unreachable and git eventually garbage-collects it. For a throwaway test task like hello.md, that's exactly what you want.
If a real task's output actually matters, merge or cherry-pick that branch into main before deleting it. Delete first, and the commit becomes unreachable — recoverable only briefly via git reflog, not something to rely on.
